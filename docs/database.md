# Аналіз нормалізації бази даних проєкту Pakin Event Scheduler

У межах проєктування схеми бази даних було проведено перевірку структури на відповідність принципам нормалізації. Метою є мінімізація над надмірності даних та забезпечення цілісності інформації.

## 1. Перша нормальна форма (1NF)

**Вимога:** Усі атрибути повинні бути атомарними (неподільними), а таблиці не повинні містити груп, що повторюються.

- **Аналіз:** Кожне поле в наших таблицях (`users`, `events`, `invitations`, `reminders`) містить лише одне значення. Наприклад, ми не зберігаємо список імейлів запрошених учасників у текстовому полі таблиці `events`. Замість цього використовується окрема сутність `invitations`, де кожен запис відповідає одному запрошенню.
- **Висновок:** Схема відповідає вимогам **1NF**.

## 2. Друга нормальна форма (2NF)

**Вимога:** Схема має відповідати 1NF та не містити часткових залежностей (кожен неключовий атрибут має залежати від усього первинного ключа).

- **Аналіз:** Усі таблиці використовують простий сурогатний ключ `id` як Primary Key. Оскільки ключ не є складеним (складається з одного поля), часткова залежність неключових полів від ключа неможлива — усі атрибути залежать від `id` в цілому.
- **Висновок:** Схема відповідає вимогам **2NF**.

## 3. Третя нормальна форма (3NF)

**Вимога:** Схема має відповідати 2NF та не містити транзитивних залежностей (неключові атрибути не повинні залежати від інших неключових атрибутів).

- **Аналіз:** \* У таблиці `events` ми зберігаємо лише `organizer_id`. Ми свідомо не додаємо поле `organizer_name`, оскільки ім'я залежить від `organizer_id`, а не від `id` події.
  - У таблиці `reminders` дані про час сповіщення (`remind_at`) та тип (`type`) стосуються конкретного факту нагадування і не залежать одне від одного.
  - Усі атрибути в таблицях безпосередньо описують сутність, ідентифіковану первинним ключем.
- **Висновок:** Схема відповідає вимогам **3NF**.

# Документація бази даних Pakin Event Scheduler

У цьому документі наведено детальний опис структури реляційної бази даних, розробленої для забезпечення функціонування системи планування подій.

## 1. Опис таблиць

### users (Користувачі)

Призначена для зберігання облікових даних та профілів користувачів системи. Основні поля включають унікальні ідентифікатори (`id`), електронну пошту (`email`), хеш пароля для безпечної автентифікації та мітку часу створення акаунта.

### events (Події)

Центральна таблиця, що містить інформацію про заплановані заходи. Вона зберігає назву події, детальний опис, часові межі (початок та кінець), локацію та посилання на організатора через `organizer_id`.

### invitations (Запрошення)

Слугує для керування доступом учасників до подій та відстеження їхніх відповідей. Таблиця фіксує зв’язок між користувачем та подією, а також поточний статус запрошення (`pending`, `accepted`, `declined`).

### reminders (Нагадування)

Призначена для планування автоматичних сповіщень. Зберігає час спрацювання (`remind_at`), тип каналу зв'язку (наприклад, email) та стан відправлення, що дозволяє системі вчасно інформувати учасників про майбутні заходи.

## 2. Опис ключових зв'язків

- **Організація подій (1:N):** Кожен запис у таблиці `events` пов'язаний з одним користувачем із таблиці `users` (організатором). Один користувач може бути власником необмеженої кількості подій.
- **Участь у подіях (M:N):** Реалізована через проміжну таблицю `invitations`. Це дозволяє одній події мати багато запрошених учасників, а одному користувачу — отримувати запрошення на багато різних подій.
- **Система сповіщень (1:N):** Кожна подія в таблиці `events` може мати декілька пов’язаних записів у таблиці `reminders`, що дозволяє налаштовувати серію нагадувань для одного заходу.
