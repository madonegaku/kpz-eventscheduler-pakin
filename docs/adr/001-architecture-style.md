# ADR-001: Вибір архітектурного стилю

## Статус: Прийнято

## Контекст

Проєкт Pakin Event Scheduler — це вебзастосунок для керування персональними та груповими календарями. Основними вимогами є висока швидкість розробки MVP, надійність зберігання даних про події та можливість реалізації складної логіки перевірки конфліктів у розкладі.

## Рішення

Використовувати **багатошарову архітектуру (Layered Architecture / MVC)**, реалізовану через фреймворк **FastAPI**. Увесь код буде розділений на три незалежні шари: Presentation (маршрути API), Business Logic (сервіси з бізнес-правилами) та Data Access (репозиторії та моделі бази даних).

## Альтернативи, що розглядалися

- **Мікросервіси:** відхилено, оскільки така архітектура потребує значних витрат на управління інфраструктурою та мережею, що є надлишковим для невеликого проєкту з одним розробником.
- **Моноліт без розділення на шари (Big Ball of Mud):** відхилено, оскільки відсутність чіткої структури ускладнить супровід коду, тестування алгоритмів планування та майбутню інтеграцію зовнішніх сервісів.

## Наслідки

- **Чіткий розподіл відповідальності:** зміни в базі даних або API-інтерфейсі не зачіпають основну бізнес-логіку системи.
- **Тестованість:** архітектура дозволяє легко писати Unit-тести для кожного шару окремо.

* **Додатковий код:** необхідність створення посередників (DTO, інтерфейси) для передачі даних між шарами дещо збільшує час на написання простих функцій.
